This guide defines how to build, test, document, and publish Terraform modules for this
repository. Follow it when creating a new module or extending an existing one.

## 1. Module Structure Template

Every module lives under `modules/<resource-type>/` and contains exactly these files:

```
modules/<resource-type>/
├── versions.tf               # Terraform and provider version constraints
├── variables.tf              # All input variables with types, descriptions, validation
├── main.tf                   # Resource definitions and locals
├── outputs.tf                # All outputs with descriptions
├── README.md                 # Auto-generated by terraform-docs (do not hand-edit body)
└── examples/
    └── basic/
        └── main.tf           # Minimal working example
```

### versions.tf

Pin the Terraform CLI floor and the AzureRM provider constraint. No other content belongs here.

```hcl
terraform {
  required_version = ">= 1.6.0"

  required_providers {
    azurerm = {
      source  = "hashicorp/azurerm"
      version = "~> 4.0"
    }
  }
}
```

The `~> 4.0` constraint allows patch and minor upgrades within the 4.x line while preventing
a breaking major-version bump from taking effect silently.

### variables.tf

Declare every input variable here. Each variable must have a `description`, an explicit `type`,
and a `validation` block where a constraint is enforceable at plan time. See
[Section 3](#3-variable-design) for full rules.

### main.tf

Contains `locals` blocks and `resource` blocks only. Do not declare variables or outputs here.
Keep locals at the top of the file so the transformation logic is visible before the resources
that use it.

```hcl
locals {
  # Derived values go here
  some_computed_value = "${var.name}-suffix"
}

resource "azurerm_example" "this" {
  # ...
}
```

Use `this` as the resource label when a module manages exactly one instance of a resource type.
Use a descriptive label only when a module manages multiple instances of the same type.

### outputs.tf

Expose every attribute that a caller might need to wire this module to another. Mark sensitive
values explicitly. See [Section 4](#4-output-design) for rules.

### examples/basic/main.tf

A standalone, self-contained example that exercises the module with the minimum required inputs
and a representative set of optional inputs. It must be valid HCL that `terraform validate`
accepts. See [Section 8](#8-writing-examples) for rules.

### README.md

The README header (title and description) is hand-authored. Everything below the
`<!-- BEGIN_TF_DOCS -->` / `<!-- END_TF_DOCS -->` markers is owned by `terraform-docs` and
must not be edited by hand. Run `make docs` to regenerate. See
[Section 10](#10-documentation-with-terraform-docs).

## 2. Naming Conventions

### Files and directories

| Item | Convention | Example |
|---|---|------|
| Module directory | `kebab-case` | `storage-account`, `key-vault` |
| Terraform files | `snake_case.tf` | `main.tf`, `versions.tf` |
| Local values | `snake_case` | `local.base_name` |
| Resource labels | `snake_case`; use `this` for singleton | `azurerm_storage_account.this` |

### Variables

- `snake_case` throughout.
- Group related variables with a common prefix that describes the subsystem:
  `network_rules_default_action`, `network_rules_ip_rules`, `network_rules_bypass`.
- Boolean variables follow the pattern `<noun>_enabled` or `<noun>_disabled`:
  `https_traffic_only_enabled`, `public_network_access_enabled`.
- Variables that accept an optional resource ID and disable a feature when empty use a
  `""` default and are documented as "Leave empty to disable":
  `log_analytics_workspace_id`, `cmk_key_vault_key_id`.

### Outputs

- `snake_case`, matching the attribute name on the underlying resource where possible:
  `id`, `name`, `primary_blob_endpoint`.
- When exposing an attribute from a nested resource, prefix with the parent:
  `resource_group_name`.

### Azure resource names

Generated by the naming module (see [Section 5](#5-using-the-naming-module)).

Pattern: `{project}-{environment}-{location_short}-{resource_abbreviation}[-{suffix}]`

Special cases handled by the naming module:
- **Storage Accounts**: no hyphens, max 24 characters, lowercase alphanumeric only, with a
  six-character hash derived from `unique_seed` appended for global uniqueness.
- **Key Vaults**: max 24 characters, alphanumeric and hyphens only.

Resource abbreviations follow Azure Cloud Adoption Framework (CAF) conventions and are
defined in `modules/naming/main.tf`. When adding a new resource type, add its abbreviation
to that map and a corresponding output to `modules/naming/outputs.tf`.

TFLint enforces `snake_case` for all Terraform identifiers via the
`terraform_naming_convention` rule in `.tflint.hcl`.

## 3. Variable Design

### Required rules

1. Every variable must have a `description` string. TFLint enforces this with
   `terraform_documented_variables`.
2. Every variable must have an explicit `type`. TFLint enforces this with
   `terraform_typed_variables`.
3. Every variable where a constraint is expressible must have a `validation` block.

### Type guidance

| Use case | Type |
|---|---|
| Free-form string | `string` |
| Enum (fixed set of values) | `string` + `validation` with `contains()` |
| True/false flag | `bool` |
| Whole number | `number` |
| Flat list of strings | `list(string)` |
| Key-value pairs | `map(string)` |
| Structured collection | `list(object({...}))` or `map(object({...}))` |

Use `optional()` within `object()` types to give object attributes defaults, avoiding the
need for callers to specify every field:

```hcl
variable "containers" {
  description = "Map of storage containers to create"
  type = map(object({
    access_type = optional(string, "private")
  }))
  default = {}
}
```

### Defaults

- Omit `default` for variables the caller must always supply (name, location, resource group).
- Provide a secure default for security-relevant variables: `public_network_access_enabled`
  defaults to `false`, `shared_access_key_enabled` defaults to `false`,
  `https_traffic_only_enabled` defaults to `true`.
- Use `""` as the default for optional resource IDs that gate conditional resources:
  `log_analytics_workspace_id = ""`.
- Use `{}` or `[]` as defaults for optional collection variables.

### Validation blocks

Write one `validation` block per constraint. Each block must have an `error_message` that
explains the allowed values concisely.

```hcl
variable "account_tier" {
  description = "Performance tier of the Storage Account (Standard or Premium)"
  type        = string
  default     = "Standard"

  validation {
    condition     = contains(["Standard", "Premium"], var.account_tier)
    error_message = "Account tier must be one of: Standard, Premium."
  }
}

variable "blob_soft_delete_retention_days" {
  description = "Number of days to retain soft-deleted blobs"
  type        = number
  default     = 30

  validation {
    condition     = var.blob_soft_delete_retention_days >= 1 && var.blob_soft_delete_retention_days <= 365
    error_message = "Blob soft delete retention days must be between 1 and 365."
  }
}
```

Use `can(regex(...))` for format validation:

```hcl
variable "name" {
  description = "Name of the Storage Account"
  type        = string

  validation {
    condition     = can(regex("^[a-z0-9]{3,24}$", var.name))
    error_message = "Storage Account name must be 3-24 characters, lowercase alphanumeric only (no hyphens)."
  }
}
```

### Tags variable

Every module that creates taggable resources must declare a `tags` variable:

```hcl
variable "tags" {
  description = "Tags to apply to all resources in this module"
  type        = map(string)
  default     = {}
}
```

See [Section 7](#7-tag-propagation-pattern) for how tags are applied.

## 4. Output Design

### What to expose

Expose every attribute a caller realistically needs to compose modules or wire resources
together. At minimum, expose:

| Attribute | Why |
|---|---|
| `id` | Required by most `depends_on` and cross-module references |
| `name` | Needed for display, logging, and passing to other Azure APIs |
| `resource_group_name` | Needed when the caller passes the group to another module |
| Endpoint URLs | Needed to configure clients (`primary_blob_endpoint`, etc.) |
| Connection strings / keys | Needed by consuming services; always mark `sensitive = true` |

### Sensitive outputs

Mark any output that contains a secret, key, connection string, or password:

```hcl
output "primary_connection_string" {
  description = "The primary connection string of the Storage Account"
  value       = azurerm_storage_account.this.primary_connection_string
  sensitive   = true
}
```

TFLint enforces that all outputs have descriptions via `terraform_documented_outputs`.

### What not to expose

Do not expose internal computed values that have no use outside the module. Do not re-expose
a variable that was passed straight through unchanged unless a caller genuinely needs it
echoed back.

## 5. Using the Naming Module

The naming module at `modules/naming/` generates CAF-compliant names for all resource types.
Consume it as a local module call at the environment or composition layer, then pass the
generated name into individual resource modules.

### Calling the naming module

```hcl
module "naming" {
  source = "../../modules/naming"

  project     = var.project       # e.g. "platform"
  environment = var.environment   # e.g. "dev"
  location    = var.location      # e.g. "eastus2"
  unique_seed = var.subscription_id  # used to derive a stable hash for globally-unique names
  suffix      = ""                # optional; appended for disambiguation
}
```

### Available name outputs

| Output | Example value |
|---|---|
| `module.naming.base_name` | `platform-dev-eus2` |
| `module.naming.resource_group` | `platform-dev-eus2-rg` |
| `module.naming.virtual_network` | `platform-dev-eus2-vnet` |
| `module.naming.subnet` | `platform-dev-eus2-snet` |
| `module.naming.network_security_group` | `platform-dev-eus2-nsg` |
| `module.naming.storage_account` | `platformdeveus2st3f8a1c` |
| `module.naming.key_vault` | `platform-dev-eus2-kv` |
| `module.naming.log_analytics_workspace` | `platform-dev-eus2-law` |
| `module.naming.managed_identity` | `platform-dev-eus2-id` |
| `module.naming.aks_cluster` | `platform-dev-eus2-aks` |

### Passing names to resource modules

```hcl
module "storage" {
  source = "../../modules/storage-account"

  name                = module.naming.storage_account
  resource_group_name = module.naming.resource_group
  location            = var.location
  tags                = local.common_tags
}
```

### Adding a new resource type to the naming module

1. Add the abbreviation to `resource_abbreviations` in `modules/naming/main.tf`.
2. Add a corresponding `output` block in `modules/naming/outputs.tf`.
3. If the resource has special character or length constraints (like storage accounts),
   add a dedicated local computation and expose it via its own output rather than using
   `standard_names`.

## 6. Adding Diagnostic Settings

Modules that create resources capable of emitting logs or metrics must include an optional
`azurerm_monitor_diagnostic_setting` resource. It is gated on the
`log_analytics_workspace_id` variable: when empty, no diagnostic setting is created.

### Variable declaration

```hcl
variable "log_analytics_workspace_id" {
  description = "Log Analytics workspace ID for diagnostic settings. Leave empty to disable diagnostics."
  type        = string
  default     = ""
}
```

### Resource pattern

```hcl
resource "azurerm_monitor_diagnostic_setting" "this" {
  count = var.log_analytics_workspace_id != "" ? 1 : 0

  name                       = "${var.name}-diag"
  target_resource_id         = azurerm_<resource_type>.this.id
  log_analytics_workspace_id = var.log_analytics_workspace_id

  # Include all log categories relevant to the resource type.
  # Use enabled_log for audit/diagnostic log categories.
  enabled_log {
    category = "AuditEvent"
  }

  # Use enabled_metric for metric categories.
  enabled_metric {
    category = "AllMetrics"
  }
}
```

### Choosing categories

Consult the Azure documentation for the specific resource type to identify available log and
metric categories. Common patterns:

| Resource type | Typical log categories | Typical metric categories |
|---|---|---|
| Key Vault | `AuditEvent`, `AzurePolicyEvaluationDetails` | `AllMetrics` |
| Storage Account | (none at account level) | `Transaction`, `Capacity` |
| AKS | `kube-apiserver`, `kube-audit`, `kube-controller-manager` | `AllMetrics` |

The diagnostic setting resource name follows the convention `${var.name}-diag`.

## 7. Tag Propagation Pattern

Tags are passed into every module as a `map(string)` variable and applied directly to
every taggable resource inside that module.

### Environment-level common tags

Define `common_tags` as a local in each environment's root module:

```hcl
locals {
  common_tags = {
    Environment = var.environment
    Project     = var.project
    ManagedBy   = "terraform"
    Repository  = "terraform-azure-project"
  }
}
```

### Passing tags into modules

Pass `common_tags` to every module call:

```hcl
module "storage" {
  source = "../../modules/storage-account"
  # ...
  tags = local.common_tags
}
```

### Merging additional tags

When a specific resource needs extra tags beyond `common_tags`, merge at the call site
rather than inside the module:

```hcl
module "storage" {
  source = "../../modules/storage-account"
  # ...
  tags = merge(local.common_tags, {
    DataClassification = "internal"
    CostCenter         = "engineering"
  })
}
```

### Applying tags inside a module

Apply `var.tags` directly to the resource without merging module-internal tags. The module
must not inject tags beyond what the caller provides:

```hcl
resource "azurerm_storage_account" "this" {
  # ...
  tags = var.tags
}
```

## 8. Writing Examples

Every module must contain `examples/basic/main.tf`. This example serves as the smoke test
and the primary usage reference for callers.

### Requirements

- The example must be self-contained. Do not reference variables or a `variables.tf`; use
  literal values.
- Supply every required variable with a realistic value.
- Demonstrate at least one non-trivial optional feature (e.g., containers, lifecycle rules,
  diagnostic settings).
- Include an `output` block that surfaces the key identifiers so the example can be
  validated with `terraform output` after apply.
- The example must pass `terraform validate` without provider configuration (use
  `required_providers` only if the example itself references a data source).

### Example structure

```hcl
# examples/basic/main.tf

module "<resource_type>" {
  source = "../../"

  # Required inputs
  name                = "<valid-example-value>"
  resource_group_name = "<rg-name>"
  location            = "eastus2"

  # Representative optional inputs
  public_network_access_enabled = false

  tags = {
    Environment = "dev"
    Project     = "myapp"
  }
}

output "<resource_type>" {
  value = {
    id   = module.<resource_type>.id
    name = module.<resource_type>.name
  }
}
```

### Storage Account example (reference)

```hcl
module "storage_account" {
  source = "../../"

  name                = "myappdeveus2sa"
  resource_group_name = "myapp-dev-eus2-rg"
  location            = "eastus2"

  public_network_access_enabled = false
  shared_access_key_enabled     = false

  containers = {
    data = { access_type = "private" }
  }

  lifecycle_rules = [
    {
      name                       = "blob-tiering"
      prefix_match               = []
      tier_to_cool_after_days    = 30
      tier_to_archive_after_days = 90
      delete_after_days          = 365
    }
  ]

  tags = {
    Environment = "dev"
    Project     = "myapp"
  }
}

output "storage_account" {
  value = {
    id                    = module.storage_account.id
    name                  = module.storage_account.name
    primary_blob_endpoint = module.storage_account.primary_blob_endpoint
  }
}
```

## 9. Testing with terraform test

Terraform native tests (`.tftest.hcl` files) live inside the module directory alongside the
source files. Run all tests for a module with:

```bash
make test-module MODULE=storage-account
# or directly:
cd modules/storage-account && terraform test
```

Run all project tests:

```bash
make test
# or directly:
terraform test -test-directory=tests
```

### Test file structure

```
modules/<resource-type>/
└── tests/
    ├── unit.tftest.hcl      # Plan-only tests; no provider credentials required
    └── integration.tftest.hcl  # Apply tests; require real Azure credentials
```

### Unit test pattern (plan-only)

Use `command = plan` for tests that validate variable defaults, locals, and computed values
without creating real resources. These can run in CI without Azure credentials.

```hcl
# tests/unit.tftest.hcl

variables {
  name                = "myappdeveus2st"
  resource_group_name = "myapp-dev-eus2-rg"
  location            = "eastus2"
}

run "defaults_are_secure" {
  command = plan

  assert {
    condition     = var.public_network_access_enabled == false
    error_message = "Public network access must default to false."
  }

  assert {
    condition     = var.shared_access_key_enabled == false
    error_message = "Shared access key must default to false."
  }

  assert {
    condition     = var.https_traffic_only_enabled == true
    error_message = "HTTPS-only must default to true."
  }
}

run "no_diagnostics_when_workspace_id_empty" {
  command = plan

  assert {
    condition     = length(azurerm_monitor_diagnostic_setting.this) == 0
    error_message = "Diagnostic setting must not be created when log_analytics_workspace_id is empty."
  }
}
```

### Integration test pattern (apply)

Use `command = apply` only for integration tests that verify real Azure resource creation.
Gate these behind a CI environment variable or a separate test suite to avoid accidental
charges during local development.

```hcl
# tests/integration.tftest.hcl

provider "azurerm" {
  features {}
}

variables {
  name                = "tftest${run.setup.outputs.unique_suffix}"
  resource_group_name = run.setup.outputs.resource_group_name
  location            = "eastus2"
}

run "setup" {
  module {
    source = "./tests/fixtures/resource-group"
  }
}

run "creates_storage_account" {
  command = apply

  assert {
    condition     = azurerm_storage_account.this.https_traffic_only_enabled == true
    error_message = "HTTPS-only enforcement must be enabled."
  }

  assert {
    condition     = azurerm_storage_account.this.min_tls_version == "TLS1_2"
    error_message = "Minimum TLS version must be TLS1_2."
  }
}
```

### What to test

| Test type | What to assert |
|---|---|
| Variable validation | That invalid inputs fail at plan time |
| Secure defaults | That security-relevant defaults have safe values |
| Conditional resources | That `count = 0` when a feature is disabled |
| Required attributes | That key resource attributes are set as expected |
| Output values | That outputs return the expected attributes |

## 10. Documentation with terraform-docs

Module READMEs are generated by `terraform-docs` using the project-level configuration at
`.terraform-docs.yml`.

### Generating docs

```bash
# Regenerate all module READMEs at once
make docs

# Regenerate a single module's README
terraform-docs markdown table --output-file README.md --output-mode inject modules/storage-account/
```

The `inject` output mode preserves everything outside the
`<!-- BEGIN_TF_DOCS -->` / `<!-- END_TF_DOCS -->` markers, so you can maintain a hand-authored
header and usage section above the generated content.

### README structure

```markdown
# Module: <resource-type>

One-paragraph description of what this module creates and its intended use.

## Usage

\```hcl
module "example" {
  source = "../path/to/module"
  # see inputs below
}
\```

<!-- BEGIN_TF_DOCS -->
<!-- terraform-docs inserts requirements, providers, resources, inputs, outputs here -->
<!-- END_TF_DOCS -->
```

### Keeping docs current

Run `make docs` before every commit that modifies `variables.tf` or `outputs.tf`. The
`.terraform-docs.yml` configuration renders:

- Requirements (Terraform version, providers)
- Providers
- Resources (with links to the AzureRM registry)
- Inputs (name, description, type, default, required)
- Outputs (name, description, sensitive flag)

The `sensitive: true` setting in `.terraform-docs.yml` ensures sensitive outputs are flagged
in the generated table.

## 11. Pre-commit Validation Checklist

Run these checks in order before opening a pull request. The `make all` target runs
fmt, lint, validate, and test in sequence.

```bash
make all
```

Or run steps individually:

### 1. Format

```bash
terraform fmt -recursive
```

All `.tf` files must be formatted per `terraform fmt` before commit. Unformatted files will
fail CI.

### 2. Lint

```bash
tflint --config=.tflint.hcl --recursive
```

The `.tflint.hcl` configuration enables:

| Rule | What it checks |
|---|---|
| `terraform_naming_convention` (snake_case) | All identifiers use snake_case |
| `terraform_documented_variables` | Every variable has a description |
| `terraform_documented_outputs` | Every output has a description |
| `terraform_typed_variables` | Every variable has an explicit type |
| AzureRM plugin rules | Resource-specific attribute validation |

Fix all warnings and errors before committing. Do not suppress rules without a comment
explaining why.

### 3. Validate

```bash
cd modules/<resource-type> && terraform init -backend=false && terraform validate
```

Validation confirms that HCL is syntactically correct and all references resolve. Run
against both the module root and the `examples/basic/` directory.

### 4. Unit tests

```bash
make test-module MODULE=<resource-type>
```

All plan-only tests must pass locally without Azure credentials.

### 5. Docs check

```bash
make docs
git diff --exit-code modules/<resource-type>/README.md
```

If the README differs from the generated output, commit the updated README.

### Complete checklist

- [ ] `terraform fmt -recursive` reports no changes
- [ ] `tflint --config=.tflint.hcl --recursive` reports no issues
- [ ] `terraform validate` passes in the module root
- [ ] `terraform validate` passes in `examples/basic/`
- [ ] All unit tests pass (`make test-module MODULE=<name>`)
- [ ] README is up to date (`make docs` produces no diff)
- [ ] Every new variable has a `description`, `type`, and `validation` block
- [ ] Every new output has a `description`
- [ ] `tags` variable is present and applied to all taggable resources
- [ ] `log_analytics_workspace_id` variable is present for resources that support diagnostics
- [ ] Resource name is sourced from the naming module or validated against naming rules

## 12. Module Versioning via Git Tags

Modules are versioned using git tags on the repository root. Tags follow Semantic Versioning
(`MAJOR.MINOR.PATCH`) with a `v` prefix.

### Versioning policy

| Change type | Version bump | Example |
|---|---|---|
| Breaking change (removed variable, changed type) | MAJOR | `v2.0.0` |
| New feature (new optional variable, new output) | MINOR | `v1.3.0` |
| Bug fix, documentation, refactor (no API change) | PATCH | `v1.2.1` |

### Creating a release tag

```bash
# Ensure main branch is clean and tests pass
git checkout main
git pull origin main
make all

# Tag the release
git tag -a v1.3.0 -m "feat(storage-account): add lifecycle rule support"
git push origin v1.3.0
```

### Referencing a module version

When modules are promoted to a remote registry or a dedicated module repository, callers
pin to a specific tag:

```hcl
module "storage" {
  source  = "git::https://github.com/org/terraform-azure-modules.git//modules/storage-account?ref=v1.3.0"
  # ...
}
```

For the current monorepo layout, modules are referenced by relative path and versioning is
implicit in the repository tag.

### Breaking change protocol

Before releasing a MAJOR version bump:

1. Open a pull request with a `BREAKING CHANGE` note in the PR description.
2. Update the module README with a migration section describing what callers must change.
3. Bump the version tag only after the PR is merged and CI is green.
4. Announce the breaking change in the repository changelog or release notes.

## 13. Creating a New Module from Scratch

Follow these steps in order when adding a module for a new Azure resource type.

### Step 1: Plan the module boundary

Decide what the module wraps. A module should map to one primary Azure resource and its
closely related supporting resources (diagnostic settings, management policies, containers).
Do not bundle unrelated resources into one module.

Answer these questions before writing code:
- What is the primary resource? (`azurerm_<resource_type>`)
- What optional sub-resources does the primary resource commonly need?
- What are the required inputs a caller must always supply?
- What security-relevant defaults should be pre-configured?
- Does this resource support diagnostic settings?
- Does this resource need a name with special character constraints?

### Step 2: Create the directory structure

```bash
MODULE=my-resource-type
mkdir -p modules/${MODULE}/examples/basic
touch modules/${MODULE}/versions.tf
touch modules/${MODULE}/variables.tf
touch modules/${MODULE}/main.tf
touch modules/${MODULE}/outputs.tf
touch modules/${MODULE}/examples/basic/main.tf
touch modules/${MODULE}/README.md
```

### Step 3: Write versions.tf

Copy the standard version pinning block:

```hcl
terraform {
  required_version = ">= 1.6.0"

  required_providers {
    azurerm = {
      source  = "hashicorp/azurerm"
      version = "~> 4.0"
    }
  }
}
```

### Step 4: Write variables.tf

Start with the required variables that every module needs:

```hcl
variable "name" {
  description = "Name of the <Resource Type>"
  type        = string

  validation {
    condition     = can(regex("<name-pattern>", var.name))
    error_message = "<Description of naming rules>."
  }
}

variable "resource_group_name" {
  description = "Name of the resource group where the <Resource Type> will be created"
  type        = string
}

variable "location" {
  description = "Azure region for the <Resource Type>"
  type        = string
}
```

Then add resource-specific variables with secure defaults where applicable. End with the
standard `tags` and `log_analytics_workspace_id` variables:

```hcl
variable "log_analytics_workspace_id" {
  description = "Log Analytics workspace ID for diagnostic settings. Leave empty to disable diagnostics."
  type        = string
  default     = ""
}

variable "tags" {
  description = "Tags to apply to the <Resource Type>"
  type        = map(string)
  default     = {}
}
```

### Step 5: Write main.tf

Define locals first, then the primary resource, then optional sub-resources:

```hcl
resource "azurerm_<resource_type>" "this" {
  name                = var.name
  resource_group_name = var.resource_group_name
  location            = var.location
  tags                = var.tags

  # ... resource-specific attributes ...
}

resource "azurerm_monitor_diagnostic_setting" "this" {
  count = var.log_analytics_workspace_id != "" ? 1 : 0

  name                       = "${var.name}-diag"
  target_resource_id         = azurerm_<resource_type>.this.id
  log_analytics_workspace_id = var.log_analytics_workspace_id

  enabled_log {
    category = "<LogCategory>"
  }

  enabled_metric {
    category = "AllMetrics"
  }
}
```

### Step 6: Write outputs.tf

Expose at minimum `id`, `name`, and `resource_group_name`. Add endpoint URLs, sensitive
keys, and other commonly referenced attributes:

```hcl
output "id" {
  description = "The ID of the <Resource Type>"
  value       = azurerm_<resource_type>.this.id
}

output "name" {
  description = "The name of the <Resource Type>"
  value       = azurerm_<resource_type>.this.name
}

output "resource_group_name" {
  description = "The name of the resource group containing the <Resource Type>"
  value       = azurerm_<resource_type>.this.resource_group_name
}
```

### Step 7: Add the resource abbreviation to the naming module

Open `modules/naming/main.tf` and add the new resource type to `resource_abbreviations`:

```hcl
resource_abbreviations = {
  # ... existing entries ...
  my_resource_type = "mrt"   # use the CAF abbreviation if one exists
}
```

Then add the corresponding output to `modules/naming/outputs.tf`:

```hcl
output "my_resource_type" {
  description = "Generated name for <Resource Type>"
  value       = local.standard_names["my_resource_type"]
}
```

Write `examples/basic/main.tf` following the pattern in [Section 8](#8-writing-examples).

like `"TODO"` or `"changeme"`.
